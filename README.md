## What It Does

GeoAnomalyMapper processes Synthetic Aperture Radar (SAR) single-look complex (SLC)
data into Doppler vibrometry maps, then uses a Physics-Informed Neural Network (PINN)
to estimate 3D subsurface wave-speed and void-probability volumes. The visible codebase
is organized around the SAR Doppler tomography workflow described in `CONTEXT.md`.

```
Sentinel-1/Umbra SLC → Doppler Vibrometry → PINN Vibro-Elastic Inversion → 3D Visualization → Anomaly Report
```

The current operating goal is to find underground objects or voids, with deeper
connected bodies ranked ahead of shallower bodies when evidence strength is comparable.
Detected anomalies are model outputs that require independent field verification; they
are not validated drill targets or guaranteed structures.

## Validation Status

This repository does **not** currently include a reproducible scientific validation
report or formal performance metrics for continental gravity/mineral prospectivity.
Earlier gravity/mineral validation claims are not supported by the files present in
this codebase and have been removed rather than repeated without evidence.

Current verification assets are limited to:

- deterministic synthetic SAR-vibrometry fixtures generated by `sar_vibrometry.py`
- a blind known-void validation harness in `blind_validation.py` that freezes
  candidate outputs before withheld-label scoring
- deterministic parameter-set hashing/approval discipline for validation runs
  and score-time holdout checks
- no-download SAR provider comparison scaffolding for Sentinel-1 C-band versus
  X-band/commercial placeholder arms
- deterministic validation report packaging with redacted artifacts, file hashes,
  evidence labels, and explicit claim boundaries
- a stable top-level `geoanomaly.py validation ...` CLI wrapper that delegates to
  the validation-first workflow
- compile/import checks for pipeline modules
- minimal unit tests for validation-critical behavior
- pipeline reports produced during configured exploration runs

The blind harness enables scientific validation against separately held labels,
including positive-site hits/misses and negative-control false positives. It does
not establish field accuracy until real holdout runs are executed and scored.
No accuracy, enrichment, specificity, or statistical-significance metric should be
inferred from this repository without a separately versioned validation dataset and
methodology.

## Technical Summary

**SAR vibrometry:** `sar_vibrometry.py` decomposes SLC data into Doppler sub-apertures,
tracks phase/frequency shifts, and exports vibration amplitude, frequency, and quality
maps.

**PINN inversion:** `pinn_vibro_inversion.py` trains a vibro-elastic PINN over a bounded
3D domain to estimate wave-speed, density-contrast, and void-probability volumes.

**Visualization and extraction:** `visualize_3d_subsurface.py` thresholds the 3D
void-probability volume, extracts connected anomaly bodies, writes reports/CSVs, and
optionally renders 3D/2D visualizations. Outputs include `deep_target_score`,
`void_evidence_score`, and `depth_priority_score` so the deepest credible void/object
candidates sort to the top. Thin connected bodies are preserved by default; optional
morphological cleanup can be enabled only when noise removal is more important than
retaining narrow tunnels or shafts.

**Batch orchestration:** `run_biondi_exploration.py` coordinates data search/download,
synthetic fallback generation, vibrometry, PINN inversion, and visualization for the
configured target regions.

**Satellite Embedding context:** `satellite_embeddings.py` processes real exported
Google Satellite Embedding / AlphaEarth 64-band GeoTIFFs for surface-condition change
context. These embeddings do not contain SAR SLC phase history and cannot replace the
Doppler vibrometry input; they are useful for annual surface-change screening, access
routes, disturbed ground, subsidence context, and false-positive review.

## Quick Start (Research Use)

Install dependencies:

```bash
pip install -r requirements.txt
```

For local release-style testing, install the validation-first CLI entry points in
editable mode. This uses `pyproject.toml` packaging metadata and does not package
`.env`, `data/`, `results_extracted/`, downloads, or generated model products:

```bash
python -m pip install -e .
geoanomaly --version
geoanomaly health --json --skip-gpu
```

Generate a deterministic synthetic SLC fixture and ground-truth metadata:

```bash
python sar_vibrometry.py synthetic --output data/vibrometry/synthetic_test.npy
```

Run the end-to-end orchestrator with synthetic fallback when SAR credentials or data are
not available:

```bash
python run_biondi_exploration.py --phase 1 --resolution quick --synthetic-fallback
```

For deep-object/void searches, use the deep profile. It expands the inversion domain
to as much as 5 km depth, uses a lower excitation frequency, and writes deeper report
slices:

```bash
python run_biondi_exploration.py --phase 3 --resolution deep
```

Visualize an existing PINN wave-speed volume:

```bash
python visualize_3d_subsurface.py --volume data/inversion_3d/outputs/wave_speed_volume.npy
```

Create an Earth Engine export script for annual Google Satellite Embedding GeoTIFFs:

```bash
python satellite_embeddings.py export-ee-script --lat 38.3512 --lon -121.986 --buffer-deg 0.02 --years 2023 2024 --out-js data/alphaearth_exports/vacaville_export.js
```

Compare two exported 64-band embedding GeoTIFFs:

```bash
python satellite_embeddings.py compare --before data/alphaearth_exports/site_2023.tif --after data/alphaearth_exports/site_2024.tif --change-out data/alphaearth_exports/site_change.tif --summary-out data/alphaearth_exports/site_change_summary.json
```

Use an embedding anomaly raster as a weak PINN surface prior during orchestration:

```bash
python run_biondi_exploration.py --phase 1 --resolution standard --use-embeddings --embedding-raster data/alphaearth_exports/site_fused_anomaly.tif --surface-prior-weight 0.15
```

Run the safe blind-validation fixture workflow without downloads or training:

```bash
python blind_validation.py validate-public --manifest validation_examples/public_manifest_fixture.json
python blind_validation.py run --manifest validation_examples/public_manifest_fixture.json --output-dir data/blind_validation/fixture_run
python blind_validation.py score --run-manifest data/blind_validation/fixture_run/run_manifest.json --labels validation_examples/withheld_labels_fixture.json --output data/blind_validation/fixture_score.json
python blind_validation.py baseline-report data/blind_validation/fixture_score.json --output-json data/blind_validation/fixture_baseline_report.json --output-text data/blind_validation/fixture_baseline_report.txt
python blind_validation.py package-report --public-manifest validation_examples/public_manifest_fixture.json --run-manifest data/blind_validation/fixture_run/run_manifest.json --score-json data/blind_validation/fixture_score.json --baseline-json data/blind_validation/fixture_baseline_report.json --baseline-text data/blind_validation/fixture_baseline_report.txt --output-dir data/blind_validation/fixture_report_package
```

Equivalent stable validation-first wrapper examples:

```bash
python geoanomaly.py commands
python geoanomaly.py health --json --skip-gpu
python geoanomaly.py validation validate-public --manifest validation_examples/public_manifest_fixture.json
python geoanomaly.py validation package-report --public-manifest validation_examples/public_manifest_fixture.json --run-manifest data/blind_validation/fixture_run/run_manifest.json --score-json data/blind_validation/fixture_score.json --output-dir data/blind_validation/fixture_report_package
```

After editable install, the same no-download fixture chain is available through
console scripts:

```bash
geoanomaly commands
geoanomaly health --json --skip-gpu
geoanomaly validation validate-public --manifest validation_examples/public_manifest_fixture.json
geoanomaly-validation run --manifest validation_examples/public_manifest_fixture.json --output-dir data/blind_validation/fixture_run
blind-validation score --run-manifest data/blind_validation/fixture_run/run_manifest.json --labels validation_examples/withheld_labels_fixture.json --output data/blind_validation/fixture_score.json
```

Create and validate a no-label validation parameter set before holdout scoring:

```bash
python blind_validation.py init-parameters --validation-id real_holdout_broad_regions_v1 --name broad_region_quick_v1 --output data/blind_validation/parameters/broad_region_quick_v1.json
python blind_validation.py validate-parameters --parameter-set data/blind_validation/parameters/broad_region_quick_v1.json --allow-templates
```

Plan no-download SAR comparison arms with public metadata only:

```bash
python blind_validation.py validate-public --manifest validation_examples/public_manifest_sar_comparison_template.json --allow-templates
```

For real known-void validation, copy `validation_examples/public_manifest_real_template.json`,
fill only public site metadata, and keep labels separate from the run stage. Real
downloads/training are opt-in only via `--execute-real`; synthetic fallback is
disabled by default and must not be used for blind real-world scoring.
See `docs/VALIDATION_FIRST_WORKFLOW.md` for the full validation-first procedure.

Run the minimal formal test suite:

```bash
python -m compileall -q blind_validation.py geoanomaly.py tests
python -m unittest discover -s tests
```

## Release Hardening and CI

The repository includes lightweight release hardening artifacts for local install,
CI, and validation-first smoke testing:

- `pyproject.toml` declares local packaging metadata and console scripts:
  `geoanomaly`, `geoanomaly-validation`, and `blind-validation`.
- `MANIFEST.in` excludes `.env`, `data/`, `downloads/`, `outputs/`, `results/`,
  `results_extracted/`, and model/data products from source distributions.
- `.github/workflows/validation-first-ci.yml` runs without repository secrets,
  sets Earthdata credential variables to empty strings, compiles key modules,
  runs `geoanomaly health --json --skip-gpu`, prints fixture commands, and runs
  `python -m unittest discover -s tests`.
- `geoanomaly health` / `geoanomaly preflight` checks Python version, core module
  import availability, declared dependency availability, data/output directory
  state, `.env` presence without printing values, free disk, optional GPU status,
  and no-download fixture/test command availability.

The health command is intentionally local and safe: it does not load withheld
labels, does not call SAR search/download APIs, and does not start PINN training.
Use `--create-dirs` only when you want it to create empty local output folders.

Recommended release smoke commands:

```bash
python -m pip install -e .
geoanomaly health --json --skip-gpu
python -m compileall -q blind_validation.py geoanomaly.py tests
python -m unittest discover -s tests
```

Real SAR downloads require NASA Earthdata credentials for ASF/Sentinel-1 access.
Copy `.env.example` to a local, untracked `.env` file and fill in your own
values:

```bash
EARTHDATA_TOKEN=
EARTHDATA_USERNAME=
EARTHDATA_PASSWORD=
```

Deployment helper scripts create empty placeholders or forward existing
`EARTHDATA_*` environment variables without displaying their values. Do not
commit `.env`, and rotate credentials if they were ever committed, shared, or
captured in logs.

Secret-handling rules for release validation:

- Do not pass `.env` files, tokens, passwords, bearer headers, private URLs, or
  credential logs to tests, CI, validation report packages, or issue/chat text.
- Health/preflight output may report that a key name is present, but must not
  print credential values.
- CI must run with empty credential environment variables unless a separate,
  explicitly reviewed real-data workflow is created later.
- Real downloads remain opt-in only through validation commands that use
  `--execute-real`; release hardening and fixture smoke tests must not use it.

## Known Limitations

- SAR Doppler tomography outputs are model-derived anomaly indicators, not confirmed
  structures.
- AlphaEarth / Google Satellite Embedding layers are surface-condition embeddings, not
  direct subsurface measurements.
- Sentinel-1 C-band penetration and TOPSAR geometry limit deep or small-feature
  interpretation.
- X-band, Umbra, Capella, ICEYE, and other commercial comparison arms are safe
  metadata scaffolds unless a concrete provider catalog/lock/download path is
  implemented and explicitly enabled; current locked real execution is
  Sentinel-1 only.
- PINN inversion is non-unique and sensitive to domain, frequency, threshold, and data
  quality assumptions.
- Synthetic fixtures verify plumbing and deterministic extraction behavior; they do not
  establish field accuracy.
- The blind validation harness enforces run/scorer separation, but real-world accuracy
  remains unknown until real holdout sites are processed and scored.
- Validation report packages are audit artifacts with redacted copied inputs and hashes;
  they still report candidate detections unless independently field-verified.
- Independent validation and ground truthing are required before operational decisions.
